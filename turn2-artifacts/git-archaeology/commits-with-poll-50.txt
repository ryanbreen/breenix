    docs(xhci): relabel poll_hid_events as load-bearing, not MSI fallback
    
    The 'Polling Mode (fallback for systems without interrupt support)' section header and 'safety net only' function doc were both misleading. Runtime evidence (turn2-runtime-evidence.md) confirms poll_hid_events() owns the ONLY path that enables the xHCI GIC SPI on aarch64 (poll >= 50 gate). Without it, handle_interrupt() can never be triggered for HID events.
    
    This commit only updates comments and docstrings -- no behavioral changes. The question of whether the event-ring drain overlap can be narrowed is explicitly deferred pending runtime evidence (blocked by pre-existing CPU0 timer regression on Parallels).
    
    Refs: turn1-xhci-dispatch-map.md, turn2-runtime-evidence.md

Author: Ryan Breen <ryan@ryanbreen.com>

    fix(usb): F32t Phase 5a move xHCI SPI enable to end of init
    
    Previously the GIC SPI for xHCI was enabled lazily inside poll_hid_events()
    when its poll counter reached 50 (~250ms after init). That coupled MSI
    activation to the timer-driven polling path.
    
    Move SPI enable to the end of xhci::init(), after:
    - PCI MSI capability programmed with Linux-order sequence (Phase 2)
    - state.irq = early_irq stored (Phase 4)
    - IMAN.IE = 1, USBCMD.RS=1 set
    - Enumeration complete, TRBs queued for HID endpoints
    
    This is where Linux enables interrupts at the end of controller bring-up
    (drivers/usb/host/xhci.c::xhci_run_finished_at). poll_hid_events() no longer
    holds the only path to activation; Phase 5b will delete that function.
    
    Validation: clean aarch64 kernel build. Runtime test pending.
    
    Co-authored-by: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
