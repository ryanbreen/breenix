🚫 Prime directive — Parallels is reference hardware. Never blame the hypervisor.

Under NO circumstances may Codex, a Ralph turn, or any Opus orchestrator declare a bug a "Parallels quirk", "HVF limitation", "hypervisor bug", or "Parallels doesn't support X." There is no such thing. Parallels implements the VirtIO / VirGL / GIC / PCI specs correctly — Linux proves this on the same machine every single time. If something does not work, it is a Breenix bug. Go find it.
The only two ways this claim may ever be made — and only with attached evidence:
  1. Reproduce-broken-on-Linux: the identical operation, exercised on the Linux probe VM (10.211.55.3, virtio_gpu/DRM), fails the same way — captured via bpftrace/ftrace/dmesg.
  2. Linux-works-around-it: Linux/Mesa source or a trace shows Linux itself special-casing the same Parallels behavior.
Absent (1) or (2), "it's Parallels" is forbidden. The mandatory move is: form the hypothesis → validate it on Linux FIRST → then fix Breenix. Reverting + moving on without root-causing is a violation.
(History: every prior "Parallels can't do X" claim was disproven — head=39 descriptors, BSS-DMA, used-ring IDs. All were Breenix bugs.)

Level-set I/O audit verdict (2026-05-29): The steady-state completion paths across GPU, network, input, storage, timer, and the scheduler wake-side are genuinely interrupt-driven / block-and-wake — no load-bearing busy-polling on the primary runtime paths. Threads truly sleep (WFI/HLT) and are woken by ISRs via lock-free wakeup buffers. The exceptions are enumerated below (E3): one Parallels-relevant fallback busy-poll and one architectural weakness (per-tick re-check in some blocking syscalls).
