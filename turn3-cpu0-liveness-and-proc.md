# Turn 3: CPU0 Liveness and /proc/xhci/counters

Status: COMPLETE

## Scope

Turn 3 had two required sub-objectives:

1. Root-cause why CPU0 stopped ticking after removing timer-driven I/O polling.
2. Add userspace-readable xHCI trace counters and prove MSI delivery from userspace.

Both sub-objectives were validated on Parallels aarch64 with the CPU0 timer I/O callouts removed.

## CPU0 liveness root cause

The aarch64 timer ISR already re-arms the virtual timer at the top of the handler. The protected arm-at-top sequence in `kernel/src/arch_impl/aarch64/timer_interrupt.rs` is the Breenix equivalent of Linux's periodic tick reprogramming path: timer interrupt arrives, the next tick is programmed before returning, and idle CPUs should wake from WFI on the next timer interrupt.

Turn 2 showed CPU0 only died after removing the timer-path I/O work. That means the device polling was not logically needed by the scheduler, but it was incidentally providing MMIO exits after every timer re-arm. On Parallels/HVF, the re-armed virtual timer is not observed reliably unless the guest exits after writing `cntv_tval_el0`/`cntv_ctl_el0`.

The fix is in `kernel/src/arch_impl/aarch64/timer.rs::arm_timer()`: after programming the virtual timer, Parallels guests perform one GICD_CTLR MMIO read. This is a single ordering/synchronization access, not device-state polling. It does not inspect xHCI/EHCI/keyboard state, does not drain rings, and does not schedule I/O work.

The timer interrupt no longer calls:

- `poll_keyboard_to_stdin()`
- `ehci::poll_keyboard()`
- `xhci::poll_hid_events()`
- timer-raised network RX softirq

## /proc/xhci/counters

Added `kernel/src/tracing/providers/xhci.rs` with lock-free counters:

- `XHCI_MSI_EVENT_TOTAL`
- `XHCI_IRQ_ENTRY_TOTAL`
- `XHCI_LOCK_CONTENDED_TOTAL`

Added `/proc/xhci/counters`, generated by `kernel/src/fs/procfs/xhci.rs`, and a userspace program at `userspace/programs/src/xhci_counters.rs` that reads the file after PID 1 starts.

The xHCI IRQ handler now increments the IRQ-entry and MSI-event counters. If the controller lock is contended, the handler increments the contention counter and re-enables the SPI before returning, so the interrupt path no longer relies on timer rescue behavior.
