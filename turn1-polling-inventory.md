## C. CPU0 Timer Regression Note

The CPU0 regression alarm itself is not a polling-elimination target: it is a timer watchdog in a gold-master region (`kernel/src/arch_impl/aarch64/timer_interrupt.rs`) and Linux has comparable watchdog/liveness checks. It is still operationally relevant because current xHCI, VirtIO input, EHCI input, and network progress depend on CPU0 timer callbacks. Removing those callbacks should reduce the system's dependence on CPU0 timer health, but runtime MSI verification still depends on CPU0's interrupt path staying alive long enough to observe trace counters. Turn 1 found no evidence that CPU0 divergence is caused by a separate device polling path beyond the timer callbacks already inventoried; root-cause should remain deferred unless MSI verification remains blocked after P1/P5/P6 are removed.

## E. Turn 2 Proposal

Turn 2 should start with P1/P5/P6 together because all three are CPU0 timer callbacks in the same hot path. The first implementation cut should remove the timer calls to xHCI, VirtIO input, EHCI input, and net softirq injection, then make xHCI IRQ lock contention non-lossy and route input/net completions through IRQ-triggered softirq/work. That is the dependency root for runtime evidence: as long as timer callbacks remain, MSI and input/network IRQ measurements are contaminated.

After that, proceed in dependency order:

1. xHCI command/recovery completions (P2/P3), because P1 deletion depends on no timer-side recovery.
2. Network RX/TX completion conversion (P6/P7/P8), because net init currently intentionally delays MSI enable.
3. Generic completion and early-boot AHCI removal (P12/P18), because this affects boot rootfs and init preload.
4. VirtIO GPU MMIO and GPU watchdog cleanup (P9/P10).
5. Legacy/secondary driver busy waits (P11/P13/P14/P15/P16/P17) plus grep-gate allowlist/removal.

## F. Honest Scope Estimate

This is not a two-turn cleanup. The verified inventory is about 18 real polling/conversion items plus several grep blockers that need removal or explicit Linux-equivalent allowlists. A realistic plan is 14-20 turns:

- 4-6 turns for xHCI timer deletion, IRQ-only event drain, command completion, endpoint recovery, and Parallels MSI runtime evidence.
- 3-4 turns for VirtIO input/EHCI input and network RX/TX/init conversion.
- 2-3 turns for AHCI early-boot completion and generic completion fallback removal.
- 2-3 turns for VirtIO GPU MMIO, GPU watchdog, SVGA/e1000 secondary drivers.
- 2-4 turns for grep-gate cleanup, Linux comparison finalization, aarch64/x86 builds, Parallels stress, and no-regression evidence.
